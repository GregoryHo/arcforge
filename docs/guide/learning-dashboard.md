# Learning Dashboard

## Overview

The learning dashboard is the primary control plane for reviewing and acting on learning candidates generated by the observer daemon. It runs as a local HTTP server (`arcforge learn dashboard`) and presents a browser-based UI at `http://localhost:3334`.

Candidates arrive in the queue after the observer daemon's LLM curator analyzes batched tool-call observations and produces structured proposals. The dashboard is where a human reviewer approves, dismisses, promotes, or deactivates each candidate — no candidate ever changes active behavior without explicit human action through these gates.

## Starting the Dashboard

```bash
arcforge learn dashboard              # default port 3334
arcforge learn dashboard --port 3335  # custom port
```

Open `http://localhost:3334` in a browser. The dashboard serves until interrupted (`Ctrl+C`).

## Candidate Lifecycle

Each candidate moves through a defined set of statuses. The dashboard enforces the Action × Status legality matrix — only valid actions for each status are shown.

| Status | Meaning |
|---|---|
| `pending_review` | Queued by LLM curator, awaiting human review |
| `needs_more_evidence` | Flagged; more observations required before approval |
| `approved` | Human-approved, ready to materialize into draft artifacts |
| `materialized` | `.draft` artifact files written, ready to activate |
| `activated` | Draft artifacts promoted to active skill/instinct files |
| `deactivated` | Previously activated candidate, now turned off |
| `dismissed` | Rejected by reviewer; no artifacts written |
| `superseded` | Replaced by an evolved candidate |

## Actions

| Action | Available From | What It Does |
|---|---|---|
| **Dismiss** | `pending_review`, `needs_more_evidence` | Closes the candidate without writing artifacts |
| **Approve** | `pending_review` | Marks the analysis as correct; enables materialization |
| **Materialize** | `approved`, `deactivated` | Writes `.draft` artifact files (inactive) |
| **Activate** | `materialized`, `deactivated` | Promotes draft artifacts to active files |
| **Promote** | `pending_review`, `approved` | Creates a new global-scope candidate from this project candidate |
| **Evolve** | `pending_review`, `approved` | Creates a revised successor candidate |
| **Deactivate** | `activated` | Turns off a previously activated candidate |

**Promote and Evolve** create a new candidate record — they do not change the source candidate's status. Promotion to global scope always requires this explicit action; silent auto-promotion is not supported.

## CLI Alternative

The dashboard UI dispatches the same actions available via CLI subcommands:

| Dashboard action | CLI equivalent |
|---|---|
| View candidates | `arcforge learn review --project` |
| Check inbox | `arcforge learn inbox --project` |
| Approve | `arcforge learn approve <id> --project` |
| Reject/Dismiss | `arcforge learn reject <id> --project` |
| Materialize | `arcforge learn materialize <id> --project` |
| Inspect | `arcforge learn inspect <id> --project` |
| List drafts | `arcforge learn drafts --project` |
| Activate | `arcforge learn activate <id> --project` |

Use `--json` on any CLI command for machine-readable output.

## Storage

```
~/.arcforge/learning/
├── candidates/
│   └── queue.jsonl         # Append-only candidate queue (Layer 5)
└── dashboard/
    └── actions.jsonl       # Append-only audit log of every dashboard action
```

The dashboard never writes active skill or instinct files directly. Materialization (Layer 7) and activation (Layer 8) are separate steps that write draft files and then promote them.

## Privacy Invariants

The dashboard wire model is allowlist-based. Raw evidence, full observation payloads, project identifiers, and internal body fields are **never served to the browser**. Only sanitized name, summary, rationale, and truncated preview windows appear in the UI. The same `sanitize-observation.js` redaction pass applied during candidate ingestion is applied again before any field is sliced into a preview.

## Security

Each dashboard session generates a per-session write token that must accompany every `POST /api/candidates/:id/action` request. The token is embedded in the page at serve time. Requests without a valid token are rejected with `403 Forbidden`.

Standard security headers (`X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`) are set on all responses.

## Common Mistakes

### Running the dashboard before enabling learning
The dashboard can open, but the candidate queue will be empty. Enable project learning first:
```bash
arcforge learn enable --project
arcforge learn status
```

### Expecting `analyze` to populate the queue
`arcforge learn analyze` was retired in v3.1. The statistical analyzer has been replaced by the observer daemon's LLM curator (Layers 3+4). Candidates are queued automatically once learning is enabled — no manual trigger is needed.

### Activating without inspecting drafts
Activation is irreversible in one step. Inspect the materialized `.draft` files before activating:
```bash
arcforge learn inspect <candidate-id> --project
arcforge learn drafts --project
```

### Confusing Approve with Activate
**Approve** only authorizes materialization — no file is written yet.
**Materialize** writes inactive `.draft` files.
**Activate** is the only action that changes active session behavior by promoting drafts to live artifacts.
