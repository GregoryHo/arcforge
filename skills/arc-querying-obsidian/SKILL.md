# arc-querying-obsidian

Knowledge Navigator — answer questions against the Obsidian vault using structured search with citations. The third leg of the Karpathy triad: Write → Query → Audit.

The core insight from Karpathy: without Query as a formal operation, insights from asking questions evaporate into chat history. With this skill, every question is answered from the vault with citations, and substantive answers can be filed back — so explorations compound just like ingested sources.

## Pipeline

Every invocation follows five steps:

```
Orient → Search → Read → Synthesize → (File Back)
```

1. **Orient** — Read `index.md` in the vault root for scope and navigation
2. **Search** — Find relevant notes using `obsidian-cli search`
3. **Read** — Drill into matching notes, extract relevant content
4. **Synthesize** — Answer with inline `[[citations]]` to wiki pages
5. **File Back** (optional) — Suggest filing substantive answers back into the vault

## Orient

Read `index.md` first — it's the vault's table of contents, organized by page type and generated by the auditor's LINT operation. This gives you a map before searching: what topics exist, what types of notes are available, where the gaps are.

If `index.md` doesn't exist, fall back to `obsidian-cli files` to list vault contents. Suggest: "Your vault has no index — run `/arc-auditing-obsidian lint` to generate one."

## Search

Use `obsidian-cli search` with a strategy based on the question type:

| Question Type | Strategy |
|---|---|
| "What do I know about X?" | Keyword search, check MOCs first for structured overviews |
| "How does X relate to Y?" | Search both terms, follow `## Relationships` sections in results |
| "What's the latest on X?" | Search + sort by `created:` date, check `log.md` for recent activity |
| "Summarize everything about X" | Start from MOC if one exists, expand to linked notes |

If search returns 20+ results, narrow by prioritizing MOCs and Syntheses over raw Sources — they contain processed knowledge, not raw material.

## Read

Drill into matching notes. Read frontmatter first to understand the note type, then content. For Synthesis notes, follow the `sources:` array to trace provenance when the user needs deeper evidence.

## Synthesize

Answer the question using vault content with **inline citations** — every key claim references the wiki page it came from:

```
According to [[Docker Setup Notes]], the team standardized on Docker Compose
for local development. [[Deployment Architecture]] adds that production uses
Kubernetes with Helm charts.
```

Output format adapts to the question:

| Question Style | Output Format |
|---|---|
| Explanation ("what is X?") | Markdown prose with inline citations |
| Comparison ("X vs Y?") | Table with cited cells |
| Timeline ("what happened with X?") | Chronological list using `created:` dates |
| Overview ("summarize X") | Structured summary with links to deeper notes |

**Vault-only answers.** Only answer from vault content — never fall back to general knowledge. If the vault has no relevant notes, say so clearly: "Your vault has no notes about X." Then suggest ingesting sources: "Want to add some sources about X via `arc-writing-obsidian`?"

This is intentional — in Karpathy's model, the vault IS the authority. Missing knowledge is a gap signal, not a fallback trigger. Gaps feed the GROW cycle.

## File Back

If the answer is substantive — a comparison, an analysis, or a discovered connection between notes — suggest filing it back into the vault:

"This answer connects several notes in a new way — want me to file it as a Synthesis note?"

This delegates to `arc-writing-obsidian` via Query-as-Ingest. The answer becomes a permanent vault artifact instead of disappearing into chat history.

Don't suggest file-back for quick lookups or simple factual retrievals — only for answers that add intellectual value beyond what any single note contains.

Always include your file-back decision at the end of the answer — either suggest filing back, or briefly explain why it's not needed (e.g., "A Synthesis note covering this already exists at [[Note]]"). This makes the decision visible to the user rather than silently skipping it.

## Session Log

After answering a query, append a structured entry to `log.md`:

```
## [YYYY-MM-DD] query | [question summary]
```

This follows the writer's dual-write pattern: `log.md` for LLM context (grep-parseable), daily notes for human browsing.

## Delegation

- Search, read, backlinks → invoke `/obsidian:obsidian-cli`
- File-back → invoke `arc-writing-obsidian` (Query-as-Ingest)
- Page type schema → defined by `arc-writing-obsidian` (source of truth)

**obsidian-cli path safety:** Use `file=` (name-based, like wikilinks) for notes with special characters (`&`, spaces, CJK). Use `path=` only for clean paths. The `&` character in filenames like "Search & Retrieval.md" breaks shell parsing with `path=`.

```
# Broken — & splits the shell command
obsidian-cli read path="Paradise/AI/RAG/Search & Retrieval.md"

# Works — file= resolves by name
obsidian-cli read file="Search & Retrieval"
```

## Vault Path

Same as sibling skills:
1. Check if `obsidian-cli` is available — ask it for the vault location
2. If Obsidian is not running, ask the user for the vault path
3. Store the path for subsequent invocations in the session
