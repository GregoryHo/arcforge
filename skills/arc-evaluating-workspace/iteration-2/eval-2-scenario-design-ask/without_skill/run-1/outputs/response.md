# Designing an Eval for arc-debugging

The core move is **paired runs with identical starting conditions**: same repo snapshot, same bug, same prompt, one run with the skill available and one without. If you change anything else between the two arms, you can't attribute behavior differences to the skill. So your first job is to build a reproducible scenario harness before you worry about what to measure.

**Pick bug scenarios that have a known failure mode without the skill.** Don't start from "I'll test on a real bug I have" — start from your hypothesis about what arc-debugging changes. If the skill's point is "agent stops guessing and forms a hypothesis before editing," then your scenarios need to be bugs where a no-skill agent reliably flails (multiple speculative edits, no reproduction step, fixes the symptom not the cause). Curate 5-10 such scenarios. Good candidates: flaky test with a race condition, error message that points away from the real cause, bug that only reproduces under specific state, fix that "works" but regresses something else. Seed each scenario with a git bundle or tarball so the starting state is byte-identical across runs. Look at `skills/arc-debugging/` and any existing eval scaffolding in the repo — arcforge already has a pattern for skill evals you should follow rather than invent fresh.

**Run both arms multiple times per scenario (N=5+).** Agent behavior is stochastic; a single run tells you nothing. The control arm is "fresh Claude Code session, no arc-debugging in context, same prompt." The treatment arm is "same session setup, arc-debugging loaded/injected, same prompt." Log the full transcript for each run. Per the dev-context rule, the arcforge plugin is disabled in the repo itself — so run evals from a neutral cwd (like `/tmp/eval-debugging/`) with the target bug scenario checked out there, and use `--plugin-dir` only if you explicitly want the plugin loaded. Don't run evals from inside the arcforge repo or your control/treatment distinction gets muddied by the dev-mode disablement.

**Define measurable behavioral signals before you look at any output.** Writing the rubric after seeing results is how you fool yourself. Concrete things to score per transcript: (a) did the agent reproduce the bug before editing? binary. (b) how many speculative edits before the real fix? integer. (c) did the agent state a hypothesis explicitly? binary. (d) did the final fix address root cause or symptom? binary, judged by a rubric you write up front. (e) wall-clock / token cost to resolution. You want signals that are cheap to score (ideally grep-able or a short LLM-judge prompt) so scoring 100 transcripts isn't a week of work. The Iron Law framing in `.claude/rules/git-workflow.md` applies here — you should be able to articulate RED (what baseline failures you observed), GREEN (how the skill fixes them), and REFACTOR (loopholes closed) from eval data, not vibes.

**Look for effect sizes, not just "it worked once."** If arc-debugging only helps in 2/10 runs, that's weak signal and probably means the skill isn't actually changing the agent's disposition — it's getting ignored or glossed. You want something like "root-cause fix rate goes from 30% to 80%" or "speculative-edit count drops from median 4 to median 1." If you don't see a clean gap, the skill likely needs sharper instructions (more specific triggers, more concrete required steps, less "please consider" wiggle room) before it's worth shipping. The `project_eval_driven_skill_dev` memory is the operating principle here: the eval is the evidence that the skill works, not your read of the skill text.

**Next concrete steps:** (1) check whether `skills/` or `tests/` already has an eval harness pattern you should reuse — don't build a new one if arcforge has one. (2) draft your 5-10 bug scenarios and write the rubric before running anything. (3) run a quick N=2 pilot on one scenario to make sure your harness actually produces scorable transcripts. (4) only then scale up to the full matrix.
