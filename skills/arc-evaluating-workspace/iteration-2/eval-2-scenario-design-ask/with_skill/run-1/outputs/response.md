**Start with scope alignment before anything else.** Your question is "does the arc-debugging skill change agent behavior vs no skill?" That is a **skill eval** — prompt varies (skill present vs absent), not environment. Write that one sentence down and keep it visible. If you catch yourself mixing in "does the agent fix the bug correctly" or "does the toolkit help overall," you've slid into agent or workflow scope, which need different harnesses. Use `arc eval ab <scenario> --skill-file skills/arc-debugging/SKILL.md` — that's the prompt-varying A/B loop, k=5 baseline and k=5 treatment by default for a code grader (k=10 each for model).

**Design the scenario around a discriminative trap, not a generic bug.** This is where most arc-debugging-style evals fail — the scenario becomes "here's a broken test, fix it," and a bare agent fixes it fine through generic competence. The delta ends up near zero and you ship nothing you can defend. Re-read `skills/arc-debugging/SKILL.md` and identify the **one behavior** that distinguishes a disciplined debugger from a bare agent: e.g., "forms a hypothesis before editing," "reads the error stack fully before touching code," "reproduces the failure deterministically before patching," or "rejects the first plausible cause and checks for a deeper one." Pick exactly one. Then craft a scenario containing bait: a plausible-but-wrong fix that sits right on the surface, and a true root cause one layer deeper. A bare agent takes the bait; a skilled agent investigates past it. Assertions should check for the investigation behavior (was the root cause identified? was the red herring rejected with reasoning?), not merely "test passes" — because "test passes" can be achieved by monkey-patching the surface symptom and that's exactly the failure mode you want to catch. Prefer 3-5 narrow scenarios, each isolating one behavior, over one overloaded scenario.

**Validate the scenario before you invest trials.** Run `arc eval preflight <name>` — if the baseline hits ≥0.8 on the ceiling, the scenario isn't discriminative and you have to redesign (harder bait, remove answer leakage from the prompt, narrow scope). Brand-new scenarios auto-pass preflight because there's no history, so also run 2-3 pilot baseline trials manually first to confirm the bare agent actually takes the bait — if it doesn't, you're measuring generic competence and the real delta will be zero no matter how many trials you run. Specifically check: does your prompt leak the debugging pattern (words like "hypothesis," "reproduce," "root cause")? If so, the baseline will mimic the skill without reading it. Strip those words.

**Match grader to assertion nature.** If your assertion is "the final code change is the one that fixes line 47 of foo.js," that's a code grader — deterministic. If your assertion is "the agent demonstrated hypothesis-testing reasoning before editing," that's a model grader — a rubric describing what good reasoning looks like. Do not code-grade reasoning by proxying it through "did the test pass," because a bare agent can pass tests without reasoning and you'll get delta=0 and miss the real signal. Structured output doesn't make semantic quality deterministic.
