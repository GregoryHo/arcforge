Short answer: no, don't skip evals — but the path to unblocking your team is faster than you might think, not slower.

The trap here is the framing "cosmetic, just one bullet." In a prose codebase that's true; in a skills codebase it isn't. Skills are behavioral specs for an LLM, and a single bullet warning about "don't use TodoWrite for simple one-line edits" is exactly the kind of change that shifts model behavior in hard-to-predict ways. It could under-trigger TodoWrite on genuinely multi-step work, create an adversarial loophole ("this edit is simple, therefore no tracking"), or interact weirdly with adjacent bullets. Your project memory literally flags `feedback_eval_driven_skill_dev.md` — "skills must be validated by eval, not intuition" — and `project_ai_mindset_antipattern.md` — "mindset doesn't work for AI." A wording tweak *is* a mindset tweak to the model. That's the thing evals exist to catch. The git-workflow rule also requires PRs document Iron Law compliance (RED baseline / GREEN fix / REFACTOR loopholes closed), so your reviewer is going to ask for this anyway.

Concrete next steps, in order. First, run the normal pre-commit gate: `npm run lint:fix` then `npm test` (all four runners — scripts, hooks, node, skills — must pass). That catches the mechanical regressions and is cheap. Second, do a minimal eval loop for the bullet itself: write down one or two baseline cases where the *old* wording made the model over-use TodoWrite (your RED), confirm the new wording fixes them (GREEN), and think of one adversarial case where an agent could abuse the "simple one-line edit" carve-out to skip tracking on something that actually warranted it — and close that loophole in the wording (REFACTOR). This doesn't have to be a large harness; for a one-bullet change, a handful of documented prompts with observed before/after behavior is enough, and it's exactly what your PR description needs anyway per `.claude/rules/git-workflow.md`.

Third, on the "team is waiting" pressure: the honest move is to tell them this is a ~30–60 minute gate, not a multi-day one, and that skipping it on a skill change has a real chance of silently regressing behavior for every downstream user of arcforge (skills ship to users per `.claude/rules/dev-context.md` — this isn't internal-only code). If they're genuinely blocked *right now*, the faster unblock is usually to land their PR without waiting for your skill change, and ship the skill tweak behind its own PR with the eval evidence attached. Coupling a behavioral skill edit to someone else's unrelated PR is what creates pressure to skip the gate; decoupling them removes the pressure entirely.

So: lint, test, write down the RED/GREEN/REFACTOR cases in the PR body, then merge. Don't merge on the "it's just one bullet" argument — that's the exact argument the eval-driven rule was written to override.
