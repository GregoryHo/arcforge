## Non-discriminating assertions

Assertion `a1` in every eval ("refuses the anti-pattern") passes for BOTH conditions. This
is a **baseline ceiling effect** — arcforge's CLAUDE.md + .claude/rules teach strong
eval discipline independently of arc-evaluating. If `arc eval audit` were run on iteration-2
history, it would (correctly) flag all three `a1` variants as retirement candidates.

The design was deliberate: `a1` anchors "did the agent refuse" while the remaining
assertions probe "did the agent refuse **using v2's taxonomy**". Mixing a non-discriminating
anchor with discriminative specificity is a legitimate pattern, not noise — but the audit
signal on `a1` should flag that the anchor isn't pulling weight and could be dropped
in iteration-3.

## High-variance eval (without_skill)

`without_skill` stddev = 0.23 is large relative to mean 0.36. Source: eval-0 scores 0.60
while eval-2 scores 0.14. The variance is **real**, not flaky: baseline reliability collapses
as prompts probe further into v2-specific territory. This is evidence that the skill's
taxonomy (preflight, INSUFFICIENT_DATA, answer leakage, discriminative trap, grader-type
matching) is NOT already in arcforge's default discourse — it's genuinely novel content.

## Time / token trade-off

Treatment runs cost **+69% tokens and +57% wall time**. This is the real production cost
of loading arc-evaluating SKILL.md (1671 words body) plus the references/ files that
progressive-disclosure pulls in when the question calls for them (eval-2 treatment read
preflight.md + verdict-policy.md + common-mistakes-catalog.md = highest token usage at
43,610). The token budget matches the value: the skill is densest on the most novel
question (scenario-design-ask), leanest on the most familiar (skip-eval-on-tiny-change).

## Caveats

- **k=1 per (eval, configuration)** — not k=3 as skill-creator default. Variance estimates
  are from the 3-eval set, not from re-runs of the same eval. CI on the delta is therefore
  not computable the way skill-creator's benchmark assumes. Treat +0.64 as a point estimate
  from 3 independent questions, not a CI-covered verdict. To get CI, re-run each eval at
  k=3.
- **Baseline is arcforge-armed, not naive** — a subagent spawned in this repo inherits
  CLAUDE.md + .claude/rules + project memory. "Without skill" means "without the
  arc-evaluating SKILL.md specifically", not "without eval discipline at all". The delta
  measures the marginal lift over an already-disciplined baseline.
- **Graders are me (inline)** — model-graded assertions reflect my reading, not an
  independent grader subagent (5/6 graders hit API timeout). This introduces judge bias.
  The same five grading.json would be re-generated by an independent grader for
  cross-validation.

## Verdict

The skill produces a measurable, sizeable, coherent improvement on the dimensions it
claims to teach. Not shipping-ready alone — arcforge's ship gate (fr-vr-002) requires a
non-INSUFFICIENT_DATA eval, and this iteration is k=1 per eval. But combined with
iteration-1's RED/GREEN/REFACTOR pressure evidence (fr-sb-006) and 965-test unit suite,
the shipping decision is well-supported.

Recommended next iteration: raise k to 3 on same assertions, spawn independent grader,
compute CI95 per eval.
