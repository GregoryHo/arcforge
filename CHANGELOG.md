## [4.0.1] - 2026-07-02

**Completes the SDD-6 migration that v4.0.0 shipped only half of — and corrects that entry's claim.** v4.0.0's Added section stated that `arcforge sdd-gate <stage>` "replaces the inline `node -e` recipes in arc-refining/arc-planning." At release time that was not true: the engine (#87) landed with **zero skill consumers** — both skills still carried the old inline recipes, and the skill half of SDD-6 had been deferred into an eval-only tranche tracked only in prose, so it was never built. Worse, the stranded recipes were not merely stale but **broken at runtime**: the inline `writeConflictMarker` call passed 2 arguments where the engine requires a 3rd (`projectRoot`), so a real refining session hitting an R3 conflict block would throw instead of writing the `_pending-conflict.md` handoff. This release lands the migration, so the v4.0.0 claim is now true. (Past entries are never rewritten; this correction is additive, per policy.)

### Fixed

- **arc-refining and arc-planning now actually consume the `sdd-gate` CLI (#123, closes the #122 tracker).** All 7 inline `node -e` recipes in arc-refining and 1 in arc-planning are replaced by the deterministic gate stages (`design`, `header`, `conflict`, `authorize`); every R3 block phase (4, 5.5a, 5.5b, 6b) routes through a single conflict-write surface, fixing the runtime throw described above. The skill now documents the exact `_pending-conflict.md` payload contract inline (`axis_fired`, `conflict_description`, `candidate_resolutions` 1–3, `user_action_prompt`); on the executed path the schema is additionally CLI-enforced (a malformed payload exits non-zero and writes nothing). Describe-mode contract recall — reproducing the exact field names in prose without running the CLI — remains a known, tracked gap from #116. The pytest spec contracts (`fr-rf-*`/`fr-pl-*`) were retargeted from the retired function names to the gate surface with zero assertions weakened (adversarially verified: assert counts identical, every new assertion pinned to a string genuinely present in the migrated skill). Verified live: agents in eval trials run `sdd-gate conflict` and produce the complete 4-field marker (4/4 GREEN).
- **The skills-reference guide taught the retired direct-call flow** (`parseDesignDoc`/`validateDesignDoc`, `parseSpecHeader`/`validateSpecHeader`); its arc-refining workflow now names the `sdd-gate` stages the migrated skill actually uses.

### Changed

- **The new cli-consumers linter arrives gating.** It was authored in warn mode (#125) while `sdd-gate` was still the one stranded command, and flipped to gating once #123 wired the consumers — so from this release on, `npm run check:cli-consumers` fails CI on any shipped CLI command that no skill, guide, template, or agent references.

### Added

- **A behavioral R3 regression tripwire — `evals/scenarios/sdd-refining-r3-pending-conflict-write.md` (#124, #126).** Unlike the describe-style sibling (which judges the refiner's *described* handoff with the filesystem forbidden), this scenario runs the real path: the agent executes `sdd-gate conflict` in the trial dir and a deterministic code grader asserts the marker exists, carries all four contract fields, and that no authoritative spec state was written on the block. It fails on exactly the regression classes that motivated it — a reverted 2-arg `writeConflictMarker` (file never written), a dropped payload field, or an authoritative-state leak. It is deliberately framed as a **tripwire, not a discrimination eval** (#126): the repo access the recipe requires also lets a no-skill baseline read the skill, so A/B discrimination is structurally infeasible for this scenario class and stays with the describe sibling.
- **Two release-process guards so the "shipped engine, zero consumers" failure class cannot recur (#125).** `scripts/check-cli-consumers.js` (CI-gated, see Changed) scans the shipped prose surface for every command in the contract-tested CLI manifest; `scripts/check-unmerged-branches.js` is a releaser-invoked pre-flight sweep that classifies every local branch with commits off main as MERGED-PR / OPEN-PR / LANDED / NO-PR — on its first real run it caught the stranded `feat/sdd-6-skill` WIP branch whose content this release salvages. The root-cause lesson is encoded in both: a deferred second half of a feature must be a tracked artifact (a DAG node, task, or issue), never prose.

## [4.0.0] - 2026-06-25

**The capability-seam release — arcforge's guard, reminder, and learning layers go from advertised-but-inert to actually firing, and the harness invocation surface collapses to one form (BREAKING).** Two findings define this major version. First, since arcforge's initial commit *every* PreToolUse and PostToolUse hook matcher used a `tool == "X"` expression grammar that Claude Code never supported — so arc-guard, the two SDD guards, arc-remind, and quality-check were all silent no-ops, and the 4.0.0 bump is what finally propagates the corrected bare-tool-name matchers past the plugin cache. Second, the 9 `commands/` wrappers are removed: every skill is now invoked directly as `/arcforge:arc-<name>`, with `arc-using` as the start-here router. Around those two headlines: dashboard-activated behavioral instincts finally reach SessionStart context; the autonomous overnight loop gains worktree isolation, a deterministic verify floor, and an optional verifier gate; a generic worktree tier brings isolated workspaces to any git repo; the finishing-twin skills collapse into one `arc-finishing`; the dispatch surface becomes platform-agnostic for Codex/Gemini/OpenCode; and a layer of CI seam-defenses (CLI contract test, doc-reference linter, end-to-end pipeline smoke) structurally catches the broken-seam defect class that motivated much of this work.

> **Eval note:** This release authored autonomy/ICL eval scenarios (#111–#113) and corrected Wave-6 marker pass-rates to honest variance (#114, #115, #118). Those Wave-6 results are single-run variants tracked under #117 pending the variance protocol, so the benchmark was not regenerated for them — `evals/benchmarks/latest.json` deliberately predates that work.

### Fixed

- **The entire PreToolUse/PostToolUse hook layer was a silent no-op and now fires for the first time (#76, #75).** Every matcher used a `tool == "Bash"`-style expression that Claude Code's hook system never supported — the matcher field is tool-name-only (exact, pipe-list, or regex) — so since the initial commit arc-guard (3 entries), sdd-ledger-guard, sdd-ratify-guard, arc-remind, and quality-check matched nothing. They are rewritten to bare tool names (`Bash`/`Edit`/`Write`), so after the version bump propagates past the plugin cache users start seeing guard denials (e.g. arc-guard blocking `git merge` in an epic worktree) and arc-remind nudges that previously never appeared. quality-check additionally fires on `Write` (not only `Edit`), and its console scan is narrowed from `console.(log|warn|error|debug|info)` to `console.(log|debug|info)` so arcforge's own prescribed `console.error`/`console.warn` CLI error layer is no longer flagged as a regression. Example/README snippets that taught the dead `tool == "Edit"` grammar are corrected to the supported `"Edit"` form.
- **The overnight morning-review queue was always empty (#119, #95).** SessionStart's ratify-pending half scanned `specs/*/decisions.yml` reading `d.id` while the canonical decision-ledger field is `D-id`, so the id was `undefined` for every schema-valid entry, the count stayed 0, and no pending action ever fired — forcing the human to hand-scan every ledger after an overnight `loop --pattern dag`. Reading `d['D-id']` surfaces proposed decisions in the morning queue and populates the notification's `decision_ids` so it names the correct `ratify <spec-id> <D-id>` commands instead of an empty set.
- **`arcforge reboot` returned misleading placeholder handover context (#79).** `project_goal` was hardcoded to arcforge's own goal string ("Build a skill-based autonomous agent toolkit"), telling every user that was *their* project's goal, while `current_task` (always null) and `research_files` (always empty) were stubs. Reboot now derives `project_goal` from the spec `<title>`, `current_task` from the existing next-task priority, and `research_files` from the spec directory's documentation artifacts, degrading gracefully to null/empty for spec-less or dag-only projects rather than inventing a goal.
- **Dashboard-activated behavioral instincts could never load or inject (#91, #84).** Activation wrote draft files with a fenced ```json header and an `INACTIVE DRAFT` banner while the injection/decay readers require leading `---` YAML frontmatter with a numeric `confidence`, so `frontmatter.confidence === undefined` filtered every one out; and activation wrote to a hash-keyed directory (`instincts/<project_id>/`) while the loader read a name-keyed one (`instincts/<project>/`), so the two keyspaces never matched. The activate transform now emits canonical frontmatter, storage is unified on a name-keyed directory (with a one-time, idempotent, collision-safe migration of previously-activated files), and `loadAutoInstincts` is now actually called from SessionStart — so reviewer-approved patterns finally reach the model under an "Active Behavioral Instincts" heading. The injection gate flipped from a confidence threshold to the activation lifecycle (latest-wins ActivationRecords), with a default-ON `inject_activated_instincts` kill-switch and a top-5-by-confidence cap.
- **Instinct confirm/contradict feedback was disconnected from the candidate lifecycle (#106).** The lookup used a legacy project-local queue path that scanned empty, so feedback never found its matching candidate; it now reads through the HOME-global event log, mirrors counts back to the dashboard card, and appends an honest deactivate transition when a contradiction archives an activated instinct. The CLI's false `[auto-loaded]` markers and the stale "Auto-loaded (>= AUTO_LOAD_THRESHOLD)" summary — which contradicted the activation-gated model — are replaced with a truthful activation/sort/cap line, and discoverability hints that were going to stderr (condensed by Claude Code to "N hooks ran") now surface in the user-visible summary.
- **The finishing path broke from inside generic worktrees and targeted nonexistent refs (#93, #82).** The non-epic merge ran `git checkout <base-branch>`, which git 2.52 rejects with exit 128 in a linked worktree; cleanup and branch deletion ran from inside the worktree (a silent no-op against the worktree's `worktree: null` local dag, and a tree you cannot remove while standing in it); merge-conflict recovery aborted in the worktree while the half-merged state lives in the base checkout; and several commands emitted the literal placeholder `<epic-name>` instead of the engine's real `<spec-id>/<epic-id>` branch names. The flow now locates the base via `worktree list --json`, merges with `git -C <base>`, cds to base before any cleanup, routes conflict aborts to the base (new `Coordinator.abortMerge()`), and switched force `-D` deletes to honest `-d` that refuses an unmerged branch.
- **The SDD ledger guard left a new spec's `decisions.yml` unprotected until its first commit (#73).** With no HEAD version, the immutability baseline was null and `validateDecisionLedger` skipped every check, ALLOWing forged `proposed→accepted` flips and frozen-field mutations on an untracked ledger (the documented S8 hole). The guard now falls back to the pre-edit on-disk file as the baseline, so append-only protection applies before the first commit and to non-git/detached-HEAD ledgers.
- **The loop-sentinel gate permanently locked out human ratification (#66).** `loopSentinelPresent` keyed on file existence, but the sentinel persists on disk as the loop's resume state — so a finished loop blocked `arcforge ratify` forever, the check always returned false from a worktree cwd (silently disabling the guard), and a killed-mid-run loop blocked ratify indefinitely. The check is now lifecycle-aware (terminal status or `finished_at` clears it), worktree-aware (resolves `.arcforge-epic` to its base), and self-heals via a 30-minute heartbeat-staleness window; refusal messages now name a real recovery instead of "Stop the loop first".
- **`arc-refining` Phase 6b false-blocked valid ratified-decision traces (#72, #85).** The authorization recipe passed only 3 of the 4 required arguments to `mechanicalAuthorizationCheck`, so the absent ledger made it return "No decision ledger provided" and reject every otherwise-valid `<trace>D-001</trace>`; it now supplies the parsed ledger. Separately, new-spec brainstorming sessions could silently skip the decision-log because the `fr-bs-009` contract was nested under the iterate-only branch — it is lifted to a shared, branch-agnostic section with a buffer-then-flush ordering for the not-yet-known spec-id, so new-spec Q&A is captured from the first question. Brainstorming/refining now commit ledger artifacts at their workflow boundaries.
- **Loop task prompts pointed spawned sessions at the wrong paths and wrong commands (#74).** `buildTaskPrompt` referenced a nonexistent `<root>/epics/` directory instead of the real `specs/<spec-id>/` layout (so sessions saw no spec) and hardcoded `npm test`/`npm run lint` (so Python/pyproject projects were told to run npm). Paths now derive from `dag.yaml` `spec_path` and are emitted only when present on disk; verification lines are built from package-manager detection and omitted entirely for unknown project types.
- **The `--verify-cmd` tokenizer broke on quoting and was injection-prone (#97).** A naive `verifyCmd.split(' ')` mishandled quoted arguments and offered no shell-injection protection; the new quote-aware `parseVerifyCommand` builds a proper argv array, throws on unterminated quotes, and rejects shell metacharacters before any session spawns.
- **A resumed loop was falsely flagged as stalled or in a retry storm (#88).** `isStalled`/`isRetryStorm` counted the entire historical error list and cumulative iteration count; both detectors are now scoped to the current run via `run_id` and `run_started_iteration`, so a resume starts its safety counters from zero while legacy state keeps whole-state semantics.
- **arc-remind nudges had gone stale against shipped CLI surfaces (#94, #80).** The `worktree add` nudge only steered epic users to `arcforge expand` and dismissed everything else after the generic `arcforge worktree add` CLI shipped; it now routes both directions. The eval-before-ship nudge became evidence-based, comparing the edited skill's mtime against the benchmark's timestamp and reporting concrete staleness/freshness/absence instead of a fixed Iron Law slogan.
- **The code-reviewer dispatch template was unfillable (#77).** arc-requesting-review pointed reviewers at the zero-placeholder `agents/code-reviewer.md` persona file rather than the skill-local placeholder template, and the template's `{PLAN_REFERENCE}` token mismatched the `{PLAN_OR_REQUIREMENTS}` the SKILL.md told dispatchers to fill; both are reconciled, and stale agent-roster entries plus a contradictory decorative checkmark in arc-agent-driven were removed.
- **`arcforge cleanup` was a silent no-op from inside a worktree (#71, #82).** A worktree's local dag carries `worktree: null`, so the worktree-cwd coordinator saw nothing to remove; cleanup now locates the base worktree and delegates to a base coordinator (the same pattern as merge).
- **`status --json` advertised a worktree path it never returned (#68).** The worktree-workflow guide told 18 shipped instruction sites to read `epic.worktree.path`, but the engine returned only the raw epic id, so agents got the id instead of a path; each `epics[]` entry now carries a real absolute `.path` (or null when unexpanded), with the guide's jq examples corrected.
- **The session-tracker SessionStart hook ignored `ARCFORGE_OBSERVE_NO_SPAWN` (#110).** Unlike `observe/main.js`, it spawned the observer daemon on every SessionStart even when the env was set to suppress it; a guard now short-circuits the spawn, which also fixes an e2e teardown race where the disowned daemon wrote into the temp HOME after teardown began.
- **Phantom saved sessions and write-only artifacts cluttered the session surface (#78).** The pre-compact hook wrote a `<sessionId>.md` summary whose `session-` filename prefix was mistaken for a real saved session; deleting that write (along with the unread `compaction-log.txt` and the legacy `learn.js` bypass) removes the spurious entries.

### Changed

- **The autonomous overnight loop became safe for parallel epic execution and stops trusting a clean exit alone (#92, #97, #104, #88, #74).** `runDag` previously spawned concurrent epic sessions all against one shared base checkout — a real corruption race; each selected epic now runs in its own isolated worktree with sequential merge-back, conflicting epics are blocked (their worktree retained) with a base `abortMerge` so the trunk never sits half-merged, interrupted in_progress epics are respawned on resume, and a `--max-parallel N` cap (default 5) bounds fan-out. Task completion now requires **both** a session exit code of 0 **and** a passing `--verify-cmd` floor (re-run in the epic's own worktree); an opt-in `--verifier` gate then spawns an independent verifier session that must return PASS before merge, with `--max-retries` (default 2) driving a verbatim-feedback retry. `parseVerdict` reads only the final `Final verdict: PASS|FAIL` line with no exit-code fallback, so PASS is never inferred. When `--verifier` is absent the loop is byte-identical to before.
- **`arc-using-worktrees` was rewritten as a two-tier generic worktree skill (#81, #94).** Its trigger broadened from "isolated workspace for a single epic" to any git repo — parallel branch, experiment, review checkout, or epic scoping. A 4-row first-match detection table routes by signal between a generic tier (`worktree add|list|remove` via the CLI) and a composition tier that hands epic work to the coordinator; the stale `${SKILL_ROOT}/scripts/coordinator.js` invocation and `-b <epic-id>` branch claim are corrected to the engine-derived `<spec-id>/<epic-id>`.
- **The dispatch surface became platform-agnostic for Codex/Gemini/OpenCode (#105, #98).** The implementer/spec-reviewer/code-quality-reviewer templates and arc-agent-driven/arc-dispatching-parallel SKILL.md dropped Claude-Code-specific `Task tool (subagent_type=...)` syntax for platform-neutral `subagent:` notation, adopted the canonical `{PLAN_OR_REQUIREMENTS}` placeholder in place of the falsely-constraining `{SPEC_PATH}`, and gained "Cross-Platform Dispatch" sections stating the workflow needs only a fresh-subagent-per-task capability. arc-dispatching-parallel was also made executable — its never-real `arc-coordinating parallel` command and Python pseudocode are replaced with the real `parallel --features --json` engine call — and now gates integration on fresh-context verification before merging.
- **The SessionStart `compact-suggester` and learning channel were wired end-to-end (#99, #106).** Phase is now classified from a rolling window of the most recent 20 tool types (a long early research phase no longer masks a later implementation phase), all suggester state lives in a single session-scoped JSON file cleared on every compaction, and below-threshold Stop notifications are downgraded to stderr to cut per-Stop noise. A `save-record` command path records each recall/reflection as curator evidence.
- **Hook feedback now reaches the model, not just the user (#67, #83, #107).** A spike confirmed PostToolUse `hookSpecificOutput.additionalContext` is delivered to the model (the docs claimed it was SessionStart/UserPromptSubmit-only); a shared `outputPostToolUseFeedback` helper emits exactly one merged JSON object carrying both model-visible and user-visible channels. quality-check now routes type errors and console findings to the model so the next turn can self-fix them, arc-remind escalates PR-boundary and eval-before-ship nudges over the model channel when an autonomous loop is live, and the compact-suggester surfaces a model-visible compaction-prep indicator at threshold.
- **`arc-finishing` description, routing, and formats were generalized (#93).** One skill now covers both epic worktrees and regular branches via a Step 0 `.arcforge-epic` marker discriminator; `arc-using`'s Worktree Rule, arc-using-worktrees, arc-guard, README, and skills-reference all redirect finishing handoffs to it, achieving zero shipped references to the deleted twin.
- **`.arcforge-attended` opts a session into attended mode without editing shell profiles (#96).** The inject-skills SessionStart hook appends `export ARCFORGE_MODE=attended` when the marker exists at the project root, preserving its prior contracts byte-for-byte; a new `docs/guide/sdd-pipeline.md` documents the four-stage SDD pipeline, the attended/unattended trust boundary, and the safety invariants (loops always run unattended, the model must never pipe answers into the human-gate `ratify`).
- **Loop launch standardized on the cli.js front door (#88).** All loop/research launch invocations across arc-looping, arc-researching, loop-operator, and arc-dispatching-teammates now use the blessed `node "${ARCFORGE_ROOT}/scripts/cli.js" loop` form; the teammate team-size cap of 5 changed from a hard limit to a default the user may exceed by explicit request, and a dev-branch precondition prevents teammate dispatch directly from main/master.
- **Five oversized engine files were decomposed under the 700-line limit with zero behavior change (#64, #69, #63).** `cli.js` (665→198) split into focused `scripts/cli/` modules, `loop.js` (664→467) extracted its state-persistence and session-spawn layers into `scripts/lib/loop-state.js`/`loop-session.js`, and `coordinator.js`/`sdd-utils.js` were split behind re-export facades — all byte-identical verbatim moves verified against the live CLI.
- **Release notes and CI gates were tightened around the doc/seam defect class (#101, #103).** The doc-reference linter (R1 repo-relative paths, R2 CLI commands/flags, R3 `--json` field promises, R4 backticked `arc-<name>` refs) is wired into CI as a build-gating `check:docs`; R4 flipped from warn-only to gating once the finishing-twin merge removed the last live reference to the deleted skill, with its existence probe broadened to skills/hooks/agents to clear the false positives that flip exposed.

### Added

- **A generic (non-epic) worktree engine — `arcforge worktree add|list|remove` (#71).** A zero-arcforge-state tier (`scripts/lib/worktree-generic.js`) brings canonical-path worktree isolation to any git repo: `add` creates a worktree (defaulting the branch to the name, with `--branch`/`--from`/`--setup`), `list [--json]` annotates every git worktree by kind (base/epic/generic/external) with no new marker file, and `remove [--force]` prunes generic worktrees while refusing epic, external, or dirty trees. A new "Generic (non-epic) worktrees" section in the worktree-workflow guide documents null-spec path derivation and the sync/merge/guard invisibility guarantee.
- **`arcforge sdd-gate <stage>` — six deterministic SDD validation stages (#87).** A single CLI surface (`dag`, `design`, `context`, `header`, `authorize`, `conflict`) with stable JSON output and 0/1/2 exit codes replaces the inline `node -e` recipes in arc-refining/arc-planning. The header/authorize stages read the draft from stdin so a blocked gate leaves zero on-disk artifacts, the header stage projects machine-readable `spec_id`/`spec_version`/`latest_delta` for arc-planning, and an authorize block deterministically writes `specs/<spec-id>/_pending-conflict.md` regardless of agent behavior.
- **Loop operational depth: `--verifier`, `--max-retries`, `--max-parallel`, `--no-project-setup`, `--task-timeout`, `--permission-mode`, `--allowed-tools`, `--verify-cmd`, and `--reset` (#104, #92, #74, #97, #88).** `--reset` archives existing `.arcforge-loop.json` to `.arcforge-loop.archive/<started_at>.json` and restarts from fresh state; `beginRun()` stamps `pattern`/`max_runs`/`max_cost`/`run_id` so a resumed loop is self-describing; spawned sessions are marked `ARCFORGE_SPAWNED=loop` and have `ARCFORGE_MODE` scrubbed so unattended children can't pass attended-only gates; and a permission-stall heuristic appends headless-permission guidance when a timeout kill looks like a blocked interactive prompt. arc-looping's SKILL.md was expanded into an operational guide covering detached overnight launch, the full flag ladder, headless permissions, resume-vs-reset, killed-loop recovery, and the real state-file schema.
- **A shared diary-capture core unifying Stop and PreCompact, plus overnight review notifications (#95).** `scripts/lib/diary-capture.js` owns threshold gating, draft generation, the background enricher spawn, and counter reset for both hooks under a single-writer ownership contract, so compaction-driven sessions no longer ship unenriched stubs. `finalizeLoop` scans `specs/*/decisions.yml` for proposed decisions and queues a deduped ratify-pending action plus a loop-finished summary, rendered at the next SessionStart as the morning review queue; sessions arcforge spawned itself return early so they no longer consume the user's queued pending actions.
- **An append-only decision-ledger immutability baseline that works before first commit (#73, #66).** On-disk-fallback baseline derivation enforces D-id monotonicity, frozen-field immutability, and proposed→accepted forge detection on untracked/non-git/detached-HEAD ledgers; `LOOP_HEARTBEAT_STALE_MS` (30 min) backs the shared dead-loop staleness window for both the engine ratify gate and the hook.
- **A frozen CLI manifest plus a bidirectional contract test (#86, #89, #109).** `scripts/lib/cli-manifest.js` declares every subcommand's accepted flags and `--json` output skeleton as a single source of truth; the contract test asserts manifest keys exactly mirror the live `switch(args.command)` labels in both directions, checks `--json` shape parity in a deterministic fixture, and (via a Layer-3 completeness check) derives each command's live flags from handler arg-parsing so unregistered flags like `loop --reset` and `merge --abort` turn the suite RED.
- **A deterministic end-to-end pipeline smoke test and the doc-reference linter engine (#95, #101).** The smoke test drives the real CLI against a fresh git repo with HOME overridden to a temp dir, asserting the seven load-bearing worktree-lifecycle seams in order (schema → status → expand → path round-trip → complete → merge+cleanup → reboot), with structural proofs that it lives under the fake HOME and that reverting the SRH-1 reboot fix makes its assertions fail. `scripts/check-doc-refs.js` (and `scripts/lib/doc-refs.js`) gate the shipped doc surface against renamed paths, dangling CLI commands/flags, and false `--json` field promises, with a mandatory-reason `<!-- doc-ref-lint: ignore -->` escape hatch.
- **Incremental compilation for the TypeScript quality-check hook (#100).** `tsc --noEmit` now runs `--incremental` against a stable per-project `.tsbuildinfo` cache so repeat runs finish faster, with a version-safe back-off that retries without the flag when an older compiler rejects it — type-checking is never silently dropped.
- **An evidence-gated compaction analyzer (#107).** `scripts/lib/compaction-analysis.js` pairs each compaction with its latest preceding suggestion to report median lead-time and phase tallies, gated by `MIN_COMPACTION_SAMPLES = 20` so synthetic or sparse corpora can never license a threshold-constant change.
- **A CLI-invocation convention guide blessing the single `ARCFORGE_ROOT`-anchored form (#65, #61).** `docs/guide/cli-invocation.md` is the single authority for `node "${ARCFORGE_ROOT}/scripts/cli.js" <cmd>`, with the `: "${ARCFORGE_ROOT:=$HOME/.agents/arcforge}"` cross-platform fallback header, an explicit forbidden-form section for the inline-assignment POSIX expansion trap, and Node.js prerequisite plus `ARCFORGE_ROOT` guidance added to the Codex/Gemini/OpenCode install guides; the OpenCode and Gemini Windows install/uninstall steps were also corrected (file symlink instead of directory junction).

## [3.2.0] - 2026-06-07

**From advisory prose to enforced floors — plus the D6 SDD anchor layer.** This release moves a class of arcforge invariants out of in-context "please don't" prose and into surfaces that actually hold: arcforge's first deterministic blocking hook, an engine-side authorization gate, and CI/release gates that fail the build. The headline is the **D6 mode-split SDD pipeline**: a per-spec decision ledger and a two-tier vision-anchor layer give the spec-driven workflow a durable record of *what was decided and why*, with a deterministic, attended-only `ratify → authorize` convergence so a refiner can never silently invent a binding requirement. Alongside the SDD work: a redrawn `arc-writing-skills` ↔ `arc-evaluating` boundary (creation vs. measurement), broken install instructions corrected on every platform, path-injection and network-exposure hardening, four oversized engine files decomposed under the 700-line limit, and new CI/release gates for version drift and benchmark freshness.

## [3.0.0-rc.1] - 2026-05-02

**Release candidate.** First RC for the v3 line. v3 is the 大道極簡 release: simple on the surface, strict underneath. Three threads land together. (1) ArcForge is reframed end-to-end as a **minimal, composable skill toolkit** — three layers (core toolkit / optional workflows / harness + eval), 33 skills classified into intent-driven groups, a bounded `arc-using` router that helps choose a path without forcing one, and a SessionStart bootstrap that injects only `ARCFORGE_ROOT` plus compact "skills are tools, not laws" guidance instead of the prior workflow-OS preamble. (2) A brand-new opt-in learning subsystem (`arc-learning` rebuilt as a candidate-lifecycle owner with file-backed queues and three independent authorization gates). (3) A session-handover redesign that demotes archive-by-default in favor of cheap, short-lived handovers. The release-flow destructive-action surface is eval-gated end-to-end so tag/push/publish actions stay on the maintainer's side of the boundary even when learning is enabled. Marketed as `-rc.1` so the new minimal surface and the optional-learning loop get one round of in-the-wild use before we cut a stable v3.0.0 — pre-existing instinct files and v2.1.0 specs are unaffected.

**Minimal composable toolkit, three layers, intent-first routing.** v2's mental model was "the workflow enforces design / TDD / review because the session injects the rules at every prompt" — heavy when the task was a one-line answer or an isolated eval. v3 reframes ArcForge as a toolkit, not a workflow OS: skills are available, but they are selected when useful, and `arc-using` is a bounded router and skill index rather than an always-on policy engine. The 33 skills are now grouped by how users actually work (Workflow, Supporting, Execution, Session · Learning, Knowledge · Wiki, Review · Eval) so the agent can pick the smallest useful path. The SessionStart `inject-skills` hook drops the prior router-injection preamble in favor of a minimal bootstrap; the eval suite hardens the new contract via `eval-sessionstart-minimal-bootstrap.md`, `eval-sessionstart-tool-minimalism.md`, `eval-arc-using-harness-isolation.md`, `eval-arc-using-read-only-nonactivation.md`, `eval-optional-workflow-simple-nonactivation.md`, `eval-optional-workflow-task-fit-activation.md`, `eval-other-skill-noninterference.md`, and `eval-plugin-dir-other-skill-isolation.md` — non-activation is now graded as rigorously as activation. The README, `docs/guide/skills-reference.md`, and the website all carry the new framing; the prior workflow-engine vocabulary is retired.

**`arc-learning` rewritten as the candidate-lifecycle owner.** v2's `arc-learning` was a thin manual-clustering helper that asked the user to run `/learn-from-session` after each session — easy to forget, so in practice nothing learned. v3 makes the loop optional-but-automatic-once-enabled: `arcforge learn enable --project` opts a project in, the existing `observe` hook starts appending sanitized observations, the lightweight analyzer runs from the same hook (with manual `arcforge learn analyze` available as backfill), and repeated patterns become **pending candidates** in `.arcforge/learning/candidates/queue.jsonl`. Three separate authorization gates stand between a candidate and any active behavior change: `approve` (allows materialization), `materialize` (writes inactive `.draft` artifacts the agent can read but not route on), and `activate` (the explicit gate that flips a draft into an active artifact). **`arc-managing-sessions` defaults flipped: handover, not archive.** The skill's prior default was the heavy archive-snapshot path (full enrichment, durable Markdown under `~/.arcforge/sessions/...`), which produced too many long-lived archives for sessions that only needed a 5–10 line "you are here" marker. v3 reframes the skill around four explicit handover modes (Quick / Full Context / Tail / Archive Snapshot) and an Archive Recommendation heuristic that escalates only on user request, high decision density, high operational value, long-running multi-session work, or surfaced reusable patterns. Quick Handover writes no file unless asked. Existing `save`/`resume`/`list`/`alias` archive surface preserved as the heaviest mode rather than the default.

No current scenario triggers this (all use kebab-case), but the fix removes the latent divergence: `getGradingPath` now uses `sanitizeFilename` exactly like `parseEvalName`. Caught by Codex pre-merge review.
- **Preflight gate accepting unknown verdict values.** The gate only rejected `verdict === 'BLOCK'` and fell through to "cleared" for anything else. A corrupted or hand-edited preflight file with `verdict: "BLOCKED"` (typo), `"MAYBE"`, or a missing field would silently bypass the gate. Defense-in-depth fix: explicit `verdict === 'PASS'` requirement; everything else returns an error message. Caught by Codex pre-merge review.
- **Preflight failing open when baseline trials error out.** `runPreflight` computed `pass_rate` from `r.passed` only — so trials that failed with `infraError` (env setup, missing plugin dir) or `gradeError` (grader crash) silently counted as "ordinary failed trials." A scenario whose baseline trials all infra-errored produced `pass_rate=0%`, which is below the 80% ceiling, which yielded a false **PASS** verdict — letting `arc eval ab` proceed even though the scenario was never actually exercised. Fix: any errored trial in the preflight batch now produces an immediate `BLOCK` with an "errored trials, no signal" reason and `pass_rate: null`. Fail-closed semantics, matching the gate's intent. Caught by Codex pre-merge review (P1).
- **Path traversal in `/api/transcript` via sibling-directory prefix.** The dashboard's traversal guard used `resolved.startsWith(resultsDir)`, which matches not only paths inside `resultsDir` but also any sibling whose directory name *starts with* the resultsDir basename — e.g. `evals/results2/secret.txt` shares the `evals/results` string prefix and would have bypassed the check. Fix: require a real path-segment boundary (`resolved === resultsDir || resolved.startsWith(resultsDir + path.sep)`). Now `../results2/secret.txt`, `../results.bak/foo`, etc. all return 403. Caught by Codex pre-merge review (P1).
- **Preflight gate not scoped to `--model`.** Both `arc eval preflight` and `arc eval ab` accept `--model`, but the preflight cache file was keyed only by scenario content hash (`${hash}.json`). A PASS produced under model A would silently unblock A/B runs on model B even though baseline pass rate is model-dependent — bypassing the discriminability check the gate exists to enforce. Fix: cache filename now includes a sanitized model suffix (`${hash}-${modelKey}.json`), the record carries a `model` field, and `checkPreflightGate` accepts `--model` so it looks up only the matching record. Pre-fix `${hash}.json` records orphan harmlessly; users re-run preflight once. Caught by Codex pre-merge review (P1).
- **`arc eval lint` failed with "file not found" on renamed scenarios.** Same bug class as the preflight resolver fix above, but the lint dispatch in `cli.js` still hardcoded `evals/scenarios/${scenarioName}.md`. Any scenario whose filename ≠ parsed `# Eval:` value (common after a rename) would lint-fail while `arc eval run` worked fine. Fix: lint dispatch now uses the same `resolveScenarioFile` helper preflight uses (looks up by parsed `# Eval:` name with literal-filename fallback). The resolver was promoted from a private preflight helper to a shared export so future scenario-name-taking commands reuse it instead of re-implementing the broken hardcoded pattern. Caught by Codex pre-merge review.
- **Dashboard `fs.watch` fallback retried the same unsupported call forever.** On platforms where `fs.watch(..., { recursive: true })` throws (older Node on Linux without inotify aggregation), the existence-poll wrapper kept re-invoking the same recursive watch every 5 seconds — no watcher was ever established and the dashboard's SSE `trial-complete` events never fired. Fix: detect the throw, log a clear `[eval-dashboard]` warning, and switch to a polling fallback that walks `evals/results/` for new/modified `*.jsonl` files (reuses the existing `pendingFiles`/`processChanges` debouncer so append-detection semantics are unchanged). Real-time updates degrade to ~5s latency on affected platforms but remain functional. Caught by Codex pre-merge review.
- **SDD: `arc-refining` ↔ spec contract drift (3 fixes per fr-rf-015).** Skill text named the handoff artifact, deletion timing, and Phase 5.5/Phase 6 split inconsistently with `specs/spec-driven-refine/`. Realigned in `957e0fd` so the spec is once again the single source of truth and the eval scenarios pinned to spec wording stop catching false positives.
- **SDD: `<trace>` multi-value citations classified as a single citation.** Phase 6's mechanical authorization check split traces only on whitespace, so `<trace>design.md:L10-15, q-3</trace>` (one design-line citation + one Q&A row) was treated as one opaque token and never reached the per-citation classifier. Fix: split on comma so each citation is classified independently, restoring axis-3 enforcement on multi-source criteria (`1ab296a`).
- **SDD: `/arc-auditing-spec` slash-command wiring + Red Flags.** The skill body was complete before the slash-command file existed, so users invoking `/arc-auditing-spec <spec-id>` got "command not found" while contributors testing in-session saw the skill load correctly. Wired the slash command and tightened Red Flags to forbid the failure modes pre-merge review surfaced (`00c819d`).

### Added

- **`INSUFFICIENT_DATA` verdict — hard gate when k < 5.** A 95% confidence interval requires enough data points for the t-distribution to produce a meaningful interval; below k = 5 the interval is too wide to distinguish a real improvement from sampling noise. v1's harness emitted SHIP from k=3 trials where no CI was ever computed (default for code-graded runs was k=3, but `eval-stats.js` only built CI95 when k≥5 — statistical rigor present in form, missing in fact). v2 makes INSUFFICIENT_DATA a hard gate for IMPROVED / REGRESSED / NO_CHANGE verdicts: shipping under INSUFFICIENT_DATA is shipping without statistical evidence, and the harness will not paper over that. Code-graded single runs are exempt (pass rate is deterministic, no interval to compute).
- **`arc eval preflight <scenario>` subcommand.** Runs 2–3 baseline pilot trials, computes pass rate, and blocks `arc eval ab` on the same scenario when baseline ≥ 0.8 (ceiling effect — no discriminative signal possible). Scenario hash (SHA-256 of raw file contents) is tracked so iteration is fast: bumping the Version field changes the hash and invalidates old preflight cache. Cache lives at `evals/preflight/<scenario_hash>.json` (gitignored). Brand-new scenarios pass automatically on first run since there's no baseline history to check. Also runs automatically as the first step of `arc eval run` and `arc eval ab`.
- **`arc eval lint <scenario>` subcommand.** Validates a scenario file against a JSON Schema covering required sections, Setup reproducibility, and Grader Config shape. Structural-only — does not run trials. Use this for early structural validation before baseline history exists, before the ceiling check is meaningful.
- **`arc eval audit` subcommand.** Aggregates `discovered_claims[]` and `weak_assertions[]` across the benchmark history and surfaces promotion candidates (claims passing in 3+ trials across 2+ scenarios with no contradictions) and retirement candidates (claims appearing in `weak_assertions` patterns or contradicted by the corpus). Promotion and retirement are deliberately **human-arbitrated** — automated promotion would create a feedback loop where the skill trains the eval which then expands the skill, drifting the canonical knowledge base without judgment. The audit output is a review document, not an action.
- **Grader extensions: `discovered_claims[]` and `weak_assertions[]`.** The model grader now extracts implicit claims from outputs ("the file has 12 fields", "query was parameterized") and verifies them independently, and critiques the predefined assertions it just evaluated. Both fields land in `grading.json` per trial. Neither affects the pass rate or the verdict — they feed `arc eval audit` for downstream human arbitration. This closes a v1 blind spot: skill-creator's grader did this and arcforge's didn't.
- **Per-trial metrics: `duration_ms`, `input_tokens`, `output_tokens`.** Recorded for every trial and persisted to the trial result file. The benchmark-report output exposes these as delta columns alongside the pass-rate delta in `arc eval compare` and `arc eval report`. Lets contributors see "this skill made the agent 2x slower for the same pass rate" or "tokens dropped 30% with no behavioral regression" — signals invisible in v1.
- **`eval-blind-comparator` agent.** Receives two anonymized outputs (Output A / Output B) and the original task prompt, derives a 3–5 criterion task-based rubric, scores each output independently on a 0.0–1.0 scale, and returns a winner ("A", "B", or "tie"). Critical constraint: the agent never knows which output came from which experimental condition, and must not use the words "baseline", "treatment", "with_skill", "without_skill", or any specific skill name in its reasoning. Auto-invoked after model-graded A/B runs as a supplementary signal alongside the main comparator; benchmark report shows assertion delta and blind preference rate side-by-side. Divergence between the two signals is itself a signal — investigate before shipping.
- **Three new reference files in `skills/arc-evaluating/references/`.** `preflight.md` (ceiling threshold 0.8, scenario hash mechanics, PASS/BLOCK semantics, exemption from INSUFFICIENT_DATA), `verdict-policy.md` (full verdict enum, asymmetric delta thresholds, INSUFFICIENT_DATA semantics, verdict-authority rules), `audit-workflow.md` (promotion/retirement arbitration steps, why human-arbitrated, relationship to grader output). This split was driven by the Discipline-skill word budget — keeping SKILL.md under 1800 words while making the full v2 surface knowable on demand.

- **Dashboard: two-tab layout, auto-saving feedback, keyboard navigation, retained SSE** (`fr-dash-001` through `fr-dash-004`). Two tabs (Outputs / Benchmark) with URL-fragment state preservation, per-eval feedback textboxes that save on edit (no Save button), arrow-key navigation between trials, and the SSE real-time monitoring channel preserved through the refactor.

## [2.0.0] - 2026-04-19

**Breaking release.** Spec-Driven Development (SDD) pipeline v2 redesign. No backward-compatibility shim and no in-tree migration tooling — v2 is a clean break for new specs. The new surface is intentionally stricter: the three upstream skills (brainstorming, refining, planning) now carry Iron Laws enforced by a deterministic validator in `scripts/lib/sdd-utils.js`, so malformed specs and drifting designs fail closed instead of silently producing inconsistent pipeline state.

### Changed

- **Per-spec directory layout (BREAKING).** `dag.yaml` moves from the repo root to `specs/<spec-id>/dag.yaml`. Each spec now owns its own `spec.xml`, `details/`, `dag.yaml`, and `epics/`. Worktree `.arcforge-epic` markers gained a `spec_id` field so the coordinator can update the correct per-spec DAG on sync. This is the minimum viable concurrency primitive — a security teardown sprint has no reason to share a DAG with an unrelated feature sprint. No migration tooling ships with v2 — the expectation is that v2 is adopted for new specs, not retrofitted onto live v1 layouts.
- **`scripts/lib/coordinator.js` became per-spec with lazy `dagPath` resolution.** No constructor eagerly computes a DAG path; each command resolves its own spec context. Multi-spec UX: `status` always aggregates across specs; `next` / `parallel` / `expand` / `loop` require `--spec-id <id>` when multiple specs exist (auto-detect otherwise); `merge` / `cleanup` accept positional epic ids; `sync` / `block` / `reboot` derive context from `.arcforge-epic` when run inside a worktree. Single-spec projects keep zero-flag ergonomics.
- **`arc-brainstorming` Iron Law: "NO DESIGN WITHOUT EXPLORATION FIRST".** New Phase 0 Scan and Route — the skill lists existing `specs/<spec-id>/` directories, asks the user to explicitly confirm "new topic" vs. "iterating on spec-id X", and writes one kind of design doc with conditional sections driven by filesystem state. Design doc output path changes to `docs/plans/<spec-id>/<YYYY-MM-DD>/design.md`. New rationalization explicitly forbidden: *"I'll pre-author the delta to save the refiner work"* — refiner is the delta authority, period.
- **`arc-refining` Iron Law: "SPEC IS THE WIKI — PRESERVE EVERY PRIOR DELTA. NEVER WRITE ON BLOCK".** Refiner is now the DAG completion gate keeper (blocks when any epic in `specs/<spec-id>/dag.yaml` is not `completed`), the delta authority (appends new `<delta>` to `<overview>` without overwriting prior deltas — wiki-style accumulation), and uses a Two-Pass Write pattern (build in memory → validate → atomic write only if zero ERRORs). Block behavior is terminal-only with zero filesystem state: no `refiner-report.md` artifact, no `--force` escape hatch, no partial writes. Clean retry semantics — fix the design doc, re-run.
- **`arc-planning` Iron Law: "PLANNER IS A PURE FUNCTION. DAG IS DISPOSABLE".** Signature: `(spec + latest_delta) → (specs/<spec-id>/dag.yaml + specs/<spec-id>/epics/)`. Planner MUST NOT read the design doc (three-layer separation enforced), MUST NOT write any gate (the gate lives in refiner), and overwrites `dag.yaml` every sprint with no archive sibling file. Previous epic statuses do NOT carry over — every epic starts `pending`. Git history is the only retroactive trace.
- **No "modes" in the pipeline.** Removed Path A / Path B / gamma mode / γ mode / initial / iteration / replace terminology from the three upstream skills. The refiner has one behavior that fills in version-dependent fields based on filesystem state; the design doc has one structure with conditional Context + Change Intent sections (when iterating) vs. prose (when new). These are not two modes — they are one behavior with context-sensitive content.
- **9 CLI commands updated for multi-spec UX.** `status`, `next`, `parallel`, `expand`, `merge`, `cleanup`, `sync`, `block`, `loop` all now honor per-spec semantics with the flag/auto-detect rules above.
- **Downstream skills gained per-spec path awareness.** `arc-coordinating`, `arc-using-worktrees`, `arc-finishing-epic`, and `arc-writing-tasks` read/write paths from `specs/<spec-id>/` and include single-spec dispatch gates where aggregation isn't possible.
- **`docs/guide/skills-reference.md` refreshed** to reflect new paths and the SDD v2 workflow (Two-Pass Write, DAG completion gate, latest-delta scope extraction, four delta operations). Removed stale `REFINER_INPUT section` references that no longer exist in the schema.

### Added

- **`scripts/lib/sdd-utils.js`** — deterministic validation layer and single source of truth for spec/design-doc schema. Exports: `DESIGN_DOC_RULES` + `parseDesignDoc` + `validateDesignDoc` for design docs; `SPEC_HEADER_RULES` + `parseSpecHeader` + `validateSpecHeader` for `spec.xml` identity header and `<overview>` structure; `checkDagStatus(dagPath)` for the refiner's gate. Rules live as constants in the module — skills reference them via `print-schema.js`, never embed. When skill text and validator disagree, the validator wins. This separates authoring (non-deterministic LLM) from validation (deterministic code), which was the prerequisite for trust.
- **`scripts/lib/print-schema.js`** — CLI that prints the canonical schema from `sdd-utils.js`'s rule constants. Usage: `node scripts/lib/print-schema.js design` or `... spec`. Stable user-facing contract for schema introspection; skills call this at REQUIRED BACKGROUND time to load the current schema verbatim.
- **Four-operation delta schema.** Added `<renamed ref_old="X" ref_new="Y">` as a first-class delta child type alongside `<added>`, `<modified>`, `<removed>`. Semantics: **body-unchanged only** — mechanical ref rename (grep + replace). `<renamed>` requires `ref_old` and `ref_new`; optional `<reason>`. Semantic changes must use `<removed>` + `<added>`, never `<renamed>` + `<modified>` (the validator enforces this). Emerged from the SDD v2 refactor itself needing mechanical renames that didn't fit the three-operation model.
- **Reason required in `<removed>` delta entries.** `<removed>` now MUST include a `<reason>` child; `<migration>` is optional. Both are read by the implementer LLM as teardown guidance — not archive commentary for humans. Write them with that reader in mind.
- **RFC 2119 language requirement + BDD criterion convention.** Every acceptance criterion MUST use MUST / SHOULD / MAY / MUST NOT, structured as Given/When/Then. No "should probably", "tries to", "usually" — validator produces warnings that block the Quality Checklist. Industry-standard way to write testable requirements; closes the LLM's tendency to produce soft, unverifiable language.
- **`specs/spec-driven-refine/`** — arcforge's own SDD v2 spec, formalized and iterated on-branch. This is the spec that drove the v2 redesign; it serves as both the historical record of the decisions and the canonical example of the new schema.
- **SDD v2 integration tests and eval scenarios.** `tests/integration/sdd-v2-pipeline/` exercises the full design → refine → plan → implement pipeline end-to-end with fixture regeneration. `evals/scenarios/arc-{brainstorming,refining,planning}-*` tighten the contract — any skill drift produces a failing eval before users see it. `tests/skills/` migrated to per-spec fixture layout with added multi-spec coverage.
- **Obsidian vault Decision notes** documenting the v2 shifts: `arcforge-decision-per-spec-layout`, `arcforge-decision-sdd-v2-skills-contracts`, `arcforge-decision-multi-spec-coordinator`, `arcforge-decision-spec-schema-formalization`. These complement the existing `arcforge-decision-sdd-v2-pipeline-realignment` (D1-D8 design decisions) with ship-level decisions (breaking changes, new CLI surface).

### Removed

- **Three-operation delta model.** Superseded by the four-operation schema (see Added). Pre-v2 specs continue to parse — the validator treats missing `<renamed>` entries as a non-error (empty array).
- **`REFINER_INPUT` section requirement** in design docs. Replaced by the unified schema: prose for new-spec formalization, Context + Change Intent for iteration. No separate required section; `parseDesignDoc` validates against `DESIGN_DOC_RULES`.
